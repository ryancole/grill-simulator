# grill-simulator

A 3D grill game built on **raw Direct3D 12**. No engine, no rendering abstraction
layer — the swapchain, command allocators and fences are all right there in
`src/renderer.cpp` on purpose.

Currently a first person camera in a walled backyard: a patio, a grill, a picnic
table, a cooler, a stack of crates and a couple of trees. Every prop is a
textured **glTF 2.0** model loaded with
[fastgltf](https://github.com/spnda/fastgltf). Only the ground, the patio slab
and the fence are still the same unit cube drawn under a different transform.

## Controls

| | |
| --- | --- |
| `Click` | capture the mouse and look around |
| `W` `A` `S` `D` | walk |
| `Shift` | sprint |
| `Space` | jump — clears 1.1 m, enough to reach the cooler and the picnic table |
| `Esc` | release the mouse, or quit if it is already free |

Anything within a 25 cm step is climbed rather than collided with, so the patio
slab is walked onto rather than bumped into.

## Requirements

| Component | Version used | Source |
| --- | --- | --- |
| LLVM / clang-cl | 22.1.8 | `winget install --id LLVM.LLVM -e` |
| MSVC toolset | 14.51 | `winget install --id Microsoft.VisualStudio.BuildTools -e --override "--quiet --wait --norestart --add Microsoft.VisualStudio.Component.VC.Tools.x86.x64"` |
| Windows SDK | 10.0.28000.0 | Standalone SDK installer, or a VS workload |
| CMake | ≥ 3.28 | `winget install --id Kitware.CMake -e` |
| Ninja | ≥ 1.13 | `winget install --id Ninja-build.Ninja -e` |

MSVC is installed for its **C++ standard library and import libraries only** —
`cl.exe` is never invoked. clang targeting the `x86_64-pc-windows-msvc` triple
does not ship its own `<vector>`, and the Windows SDK provides the C runtime but
not the C++ one, so the MSVC STL is a hard requirement for this toolchain.

The Agility SDK, DXC and fastgltf are **not** prerequisites. CMake fetches all
three at configure time and pins them by SHA-512.

Python is not a prerequisite either. It regenerates the models under `assets/`,
which are committed — see [Models](#models).

## Building

```powershell
cmake --preset windows-clang     # configure once
cmake --build --preset debug     # or: --preset release
```

Output lands in `build/bin/<Config>/`:

```
grill.exe
D3D12/          D3D12Core.dll, d3d12SDKLayers.dll   (Agility SDK runtime)
shaders/        scene.vs.cso, scene.ps.cso          (Shader Model 6.6)
assets/models/  grill.glb, tree.glb, table.glb      (binary glTF 2.0)
```

Both configurations are generated by the single configure step (`Ninja
Multi-Config`), so switching between Debug and Release never reconfigures.

## Cleaning

Every generated artifact is nested under `build/`, so deleting that one
directory is a complete reset:

```powershell
Remove-Item -Recurse -Force build
```

That covers the CMake cache, the Ninja graph, both configurations' objects and
binaries, the compiled shaders and their depfiles, clangd's background index
(`build/.cache`) and the fetched packages (`build/_deps`). Nothing generated is
written outside it.

The one cost is that `build/_deps` holds the pinned Agility SDK, DXC and
fastgltf packages, so a clean wipe re-downloads roughly 85 MB on the next
configure. To keep them across wipes, point CMake at a base directory of your
own:

```powershell
cmake --preset windows-clang -D FETCHCONTENT_BASE_DIR=C:/some/shared/cache
```

## Editing

Open the folder in VS Code and accept the recommended extensions. IntelliSense
is **clangd**, not cpptools — cpptools is installed purely for its `cppvsdbg`
debugger, and its IntelliSense engine is disabled in `.vscode/settings.json` so
the two do not produce duplicate squiggles over the same file.

Configure the project at least once before opening a source file: clangd reads
`build/compile_commands.json`, and that file is what tells it where the STL and
the Windows SDK live. Without it, every `#include` will be red.

`F5` builds Debug and launches under the Windows debugger.

## How the toolchain finds MSVC

clang's built-in Visual Studio detection only probes VS 8/9/10-era paths. It does
not find Visual Studio 18 (2026), so left to itself clang-cl locates no standard
library at all and fails on `#include <vector>`.

Rather than priming every shell with `vcvars64.bat`,
`cmake/toolchains/clang-cl-x64.cmake` discovers the roots itself (via `vswhere`
and the `KitsRoot10` registry value) and passes them explicitly:

```
-vctoolsdir <...>/VC/Tools/MSVC/<ver>  -winsdkdir <...>/Windows Kits/10  -winsdkversion <ver>
```

Two consequences worth knowing:

- Those flags land in `compile_commands.json`, which is precisely why clangd
  works with no environment set up. A `vcvars`-based build would leave clangd
  blind unless VS Code itself were launched from a developer prompt.
- CMake drives `lld-link.exe` directly rather than through the clang driver, so
  the driver's implicit `/libpath:` arguments are never generated. The toolchain
  file supplies them by hand.

Override the auto-detection with `-DMSVC_TOOLS_DIR=...`, `-DWINDOWS_SDK_DIR=...`
or `-DWINDOWS_SDK_VERSION=...` if you need a specific toolset.

## Upgrading the Agility SDK

The Agility SDK decouples the D3D12 runtime from Windows: the game ships
`D3D12Core.dll` and opts in by exporting `D3D12SDKVersion` and `D3D12SDKPath`
(see `src/agility_exports.cpp`). The system `d3d12.dll` reads those exports and
loads our redistributable core instead of the one that came with the OS.

To move to a new release, edit `cmake/AgilitySDK.cmake`:

1. Bump `AGILITY_SDK_VERSION` to the NuGet package version.
2. Bump `AGILITY_SDK_INTERFACE_VERSION` to its **minor** component —
   `1.619.4` → `619`. A mismatch between this number and the staged
   `D3D12Core.dll` fails at runtime, not at build time.
3. Replace `URL_HASH` with the new package's SHA-512.

Preview packages (`1.7xx.x-preview`) additionally require Windows **Developer
Mode** to be enabled on every machine that runs the executable. The pinned
`1.619.4` is the latest stable release and carries no such requirement.

`cmake/Shaders.cmake` pins DXC the same way.

## Upgrading fastgltf

`cmake/FastGltf.cmake` pins the release tarball by SHA-512, and then pins
fastgltf's own dependency by hand. fastgltf fetches simdjson's single-header
build with a bare `file(DOWNLOAD)` and no hash; rather than accept an unpinned
byte in the tree, we drop the two files into `deps/simdjson/` — the exact path
it looks in — *before* its configure runs. It finds them, sees a version no
older than the one it wants, and skips its own download.

So a version bump is two edits, not one:

1. Bump `FASTGLTF_VERSION` and its `URL_HASH`.
2. Check whether `SIMDJSON_TARGET_VERSION` moved in the new release's
   `cmake/dependencies.cmake`. If it did, bump `SIMDJSON_VERSION` and both
   `EXPECTED_HASH` values to match.

If step 2 is skipped and fastgltf now wants a *newer* simdjson than we staged,
it will quietly overwrite our pinned copy with an unpinned download. Nothing
fails; the pin just stops meaning anything. Watch the configure log for
`fastgltf: Found simdjson (Version ...)` — if it says `downloading...` instead,
the pin has slipped.

## Models

Props that deserve real geometry are **glTF 2.0**, loaded with fastgltf and
decoded through the Windows Imaging Component:

| Asset | Nodes | Textures | Placed |
| --- | --- | --- | --- |
| `grill.glb` | four legs, a body, a lid, a side shelf | brushed charcoal on the body | once |
| `tree.glb` | a trunk, a canopy | bark, leaves | twice, at different scale and yaw |
| `table.glb` | a top, four legs, two benches | planks | once |
| `crate.glb` | one box | cardboard | twice, stacked, the upper one askew |
| `cooler.glb` | a body, a decorative lid | moulded plastic, in two colours | once |

The ground, the patio slab and the fence are still the unit cube under a
transform. A prop crosses over when someone has something better to put there.

The committed `.glb` files are written by `tools/gen_models.py`, which builds
them out of the same boxes the code used to. It is deliberately **not** wired
into CMake — compiling a C++ game should not require a Python interpreter — so
run it by hand and commit what it writes:

```powershell
python tools/gen_models.py
```

It uses nothing but the standard library: `zlib` writes the PNGs, `struct`
writes the GLBs. The point of it is that these props now arrive through the real
asset pipeline, so swapping one for something exported from Blender is a file
copy rather than a code change.

Two things the generator does that are worth keeping if you replace it:

- **Every texture is normalised to average exactly the flat colour it
  replaced.** `shaded_texture` divides the whole brightness field by its own
  mean. That is what lets the yard read the same from across it and gain grain
  only up close — and it is what makes "did anything else change?" a question a
  pixel diff can answer.
- **UVs are laid out per metre, not per face.** `box(size, tile)` scales each
  face's UVs by its own extent, so the table's legs are cut from the same size
  boards as its top. Passing no `tile` gives the unit square, which is what the
  grill's body wants.

### One tree, twice

`tree.glb` is placed twice, at scale 1.0 with a 25° yaw and at scale 0.87 with a
−15° yaw. The canopy also carries a 12° yaw *inside* the model, so neither tree's
leaves line up with its own trunk and the two do not line up with each other.

That in-model rotation is the only one in any of these files, and it is there on
purpose: it is what exercises the loader's TRS-rotation path, including the fact
that a yaw survives the mirror through Z **with its sign flipped**. A glTF yaw of
−12° is a game yaw of +12°, which is why `yaw_quaternion()` negates its argument.

The cost of yawing a whole tree is that its trunk's collider is the world bound
of a box that is no longer axis aligned — 66 cm across rather than 50. Nobody has
ever noticed a tree being slightly too solid. Both canopies still start well
above eye height (2.80 m and 2.44 m against 1.70 m), so the player still walks
underneath them.

**glTF is right-handed with `-Z` forward; this game is left-handed with `+Z`
north.** The two differ by a mirror through Z, and `LoadGltfModel` undoes it on
the way in so nothing downstream has to know. That mirror costs three things,
and all three are in `src/model.cpp`:

- positions and normals have their Z negated,
- a node transform `M` becomes `S*M*S`,
- **every triangle is reversed**, because a mirror negates
  `cross(v1 - v0, v2 - v0)` while the outward normal it should equal only has
  its Z flipped.

One conversion is worth calling out because it looks like a bug. fastgltf stores
a matrix as 16 **column-major** floats and `XMFLOAT4X4` reads 16 floats as rows,
so a plain `memcpy` between them *is* a transpose — which is exactly what turns
glTF's column-vector convention (`p' = M*p`) into DirectXMath's row-vector one
(`p' = p*M`). Adding an `XMMatrixTranspose` there is the bug, not the fix.

### What survives the trip

A glTF material describes a full metallic-roughness BRDF. This renderer has a
hemisphere ambient, one directional light and no specular term at all, so only
`baseColorFactor` and `baseColorTexture` are read. Metallic, roughness, normal
maps and emissive are parsed past and dropped.

Skins, joints, inverse bind matrices and `JOINTS_0` / `WEIGHTS_0` **are** parsed,
into `Model::skins` and `Model::skin_vertices`. Nothing reads them yet: there is
no animation and no rigged asset. They are kept out of `Vertex` and held in a
parallel array, so that when rigging lands they become a second vertex stream
bound to slot 1 and the yard — which will never be skinned — keeps paying
nothing for them.

Colliders come from the `POSITION` accessor's `min`/`max`, which glTF requires
every asset to carry, so no vertex is ever touched to compute one. Each node
yields its own box, so the grill collides as a leg, body, lid and shelf rather
than as one loose box around the whole thing.

### Decorative nodes, and why they had to exist

**A node whose name ends in `_nocollide` is drawn and never collided with.**

One box per node is exactly what a prop assembled from parts wants — until two
of those parts are *stacked*. The player's capsule controller stands on any box's
top face, and the lower part's top face is buried inside the prop where nobody
can see it.

The cooler is where this bit. Building it as a short body under a lid put a
standable surface at 0.45 m, inside the cooler, at exactly the height of the lid.
A player descending past its corner landed on that surface and hovered there.
Sweeping every reachable state — every position around the cooler, every falling
approach — turned up tens of thousands of spots where the player caught on that
buried face. Insetting the body so the lid overhangs it only reduced the count;
nothing short of *one collider* fixes it, because there is no inset that puts a
lower box's top out of reach in every case.

So the cooler's body is the whole cooler — the same 0.9 × 0.6 × 0.6 box the yard
used to draw, and the only thing that collides — and the lid is a lip that laps
1 cm over its sides and stands 5 mm proud of its top, so the two share no face
and cannot z-fight. Collision is unchanged because the collider set is literally
unchanged, not because an argument says so.

glTF has no standard way to mark geometry decorative. A name suffix is what
engines settle on (Unreal's `UCX_`, Godot's `-noimp`) and it survives a round
trip through Blender's outliner. The alternative, a custom `extras` field, would
drag simdjson's headers into `model.cpp` to read one bool.

## How a frame is drawn

Every model — the unit cube built in code, the grill read from disk — lands in
its own default-heap vertex and index buffer, uploaded once through a staging
copy. The scene is then a flat list of **instances**: a model index, a transform,
and a tint that multiplies whatever base colour the material already had. One
draw goes out per primitive per instance.

Everything a draw needs arrives as **root constants** — 52 DWORDs written
straight into the command list — except the base colour texture, which needs a
descriptor table. The sampler is baked into the root signature, because every
texture in the game wants the same one.

Four conventions are worth knowing before editing any of this:

- **Matrices are `row_major` in HLSL and multiplied as `mul(v, M)`.** DirectXMath
  is row-major and treats vectors as rows, so declaring the HLSL side to match
  removes the transpose that samples usually do on the CPU.
- **A triangle is front facing when `cross(v1 - v0, v2 - v0)` points at the
  camera.** `MakeUnitCubeModel` emits each face's corners in the order that makes
  that cross product the outward normal, and `LoadGltfModel` reverses every
  triangle it reads to preserve the same invariant. That is why back-face culling
  works without anyone hand-checking a winding order.
- **A material with no texture samples a 1×1 white texel.** Slot 0 of the one
  descriptor heap holds it. That is what spares the pixel shader a branch and the
  pipeline a second variant, and it is why the cube's flat colours and the
  grill's textured ones go down the same path.
- **Nothing is ever converted to linear light.** The flat colours in `scene.cpp`,
  the constants in the shader, the textures (`R8G8B8A8_UNORM`, not `_SRGB`), the
  box filter that builds the mips, and the back buffer are all in whatever space
  they were written. Decoding just the textures would leave them the only linear
  thing in the frame, and they would read darker than the flat colour beside
  them. Fixing it is one change that touches all five.

Normals are carried to world space by `transpose(inverse(model))`, passed as
three `float4` rows. It matters: the yard's ground is a cube scaled 60 × 0.3 × 60,
and a model matrix that scales unevenly does not carry a normal.

## Sharp edges found the hard way

- **dxc silently skips codegen when given `-MF`.** `dxc -MF dep.d -Fo out.cso`
  exits 0, writes the dependency file, and produces no `.cso`. `add_hlsl_shader`
  therefore invokes dxc twice: once to scan, once to compile.
- **Semicolons inside a `$<IF:...>` generator expression tear it apart**, because
  CMake treats them as list separators before the expression is evaluated. Use
  one generator expression per flag.
- **`add_custom_command(OUTPUT ...)` cannot use `$<TARGET_FILE_DIR:...>`,** since
  outputs are resolved before targets exist. Hence the explicit
  `CMAKE_RUNTIME_OUTPUT_DIRECTORY`, in `add_hlsl_shader` and `stage_assets` both.
- **fastgltf downloads simdjson without a hash.** `cmake/FastGltf.cmake` fetches
  fastgltf with `SOURCE_SUBDIR does-not-exist` so `FetchContent_MakeAvailable`
  populates it without configuring it, drops a pinned simdjson into
  `deps/simdjson/`, and only then calls `add_subdirectory`.
- **`memcpy`ing a fastgltf matrix into an `XMFLOAT4X4` is the handedness
  conversion, not a bug.** Column-major in, row-major out, which is the transpose
  that carries `p' = M*p` to `p' = p*M`. See [Models](#models).
- **Stacking two colliding nodes buries a standable surface inside a prop.**
  The player's capsule stands on any box's top face, seen or not. Hence
  `_nocollide`.
- **`PW_RENDERFULLCONTENT` is mandatory to screenshot this window.** A flip-model
  swapchain prints solid black under a plain `PrintWindow`.
